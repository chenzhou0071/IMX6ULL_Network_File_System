# 嵌入式设备管理器 - 项目设计文档

**版本**: 1.0
**日期**: 2026-04-01
**项目目标**: 为NXP IMX6ULL开发板构建网络文件管理系统,作为简历项目展示后端/嵌入式应用能力

---

## 一、项目概述

### 1.1 项目背景

本项目旨在基于正点原子NXP IMX6ULL (ARM Linux)开发板,构建一个类似Samba的网络文件管理系统。电脑与开发板在同一局域网内可通过网络访问开发板的文件系统,实现文件管理和系统监控功能。

### 1.2 核心价值

- **简历亮点**: 展示网络编程、跨平台开发、软件架构能力
- **技术栈**: C语言服务器 + QT C++客户端
- **目标用户**: 开发调试、产品演示

### 1.3 核心技术能力

按优先级排序:
1. 网络编程 (Socket、TCP/IP、协议设计)
2. 跨平台开发 (QT、Windows/Linux双平台)
3. 软件架构 (模块化设计、API设计、客户端-服务器架构)
4. 系统编程 (多线程、文件系统、进程管理)

## 三、网络协议设计

### 3.1 协议特点

- **轻量级**: 适合嵌入式设备资源受限环境
- **二进制格式**: 高效传输,减少带宽占用
- **自定义设计**: 展示协议设计能力

### 3.2 协议帧格式

```
┌─────────────────────────────────────────────────────────┐
│  字段          │  大小    │  说明                       │
├─────────────────────────────────────────────────────────┤
│  magic         │  4字节   │  魔数 0xA5A5A5A5            │
│  length        │  4字节   │  数据长度(不含头部)         │
│  cmd           │  2字节   │  命令类型                   │
│  seq           │  2字节   │  序列号(请求-响应匹配)     │
│  checksum      │  4字节   │  校验和(CRC32)             │
│  data          │  变长    │  数据载荷                   │
└─────────────────────────────────────────────────────────┘
```

- **头部固定长度**: 16字节
- **字节序**: 小端序
- **最大帧长度**: 64KB

### 3.3 命令定义

#### 连接管理命令 (0x01xx)

### 3.6 协议特性

1. **魔数验证**: 快速识别协议帧,防止数据污染
2. **序列号机制**: 请求-响应匹配,支持异步操作
3. **CRC32校验**: 检测传输错误,保证数据完整性
4. **粘包处理**: 固定头部+明确长度字段
5. **心跳机制**: 30秒一次,检测连接存活

### 4.2 多线程设计

**线程类型**:
- **主线程(监听线程)**: 负责bind/listen/accept,接受新连接
- **工作线程池**: 处理客户端请求,固定数量(如4-8个)
- **监控线程**: 检查超时连接、更新系统信息缓存、写日志

**线程池**:
```c
typedef struct thread_pool {
    pthread_t* workers;              // 工作线程数组
    int worker_count;                // 线程数量
    task_queue_t* task_queue;        // 任务队列
    pthread_mutex_t queue_lock;      // 队列锁
    pthread_cond_t queue_cond;       // 队列条件变量
    bool shutdown;                   // 关闭标志
} thread_pool_t;
```

### 4.4 权限与账号管理

#### 4.4.1 配置文件格式 (server.conf)

服务器使用配置文件管理用户账号,类似Samba风格:

```ini
# 服务器配置
[server]
port = 8888
max_connections = 10
timeout = 300
log_level = info

# 用户配置
[users]
# 格式: 用户名:密码:权限(rw=读写, ro=只读)
admin:admin123:rw
guest:guest123:ro
user1:pass123:ro
```

#### 4.4.2 权限等级

```c
typedef enum {
    PERM_RO = 0,                    // 只读
    PERM_RW = 1                     // 读写
} permission_level_t;
```

| 权限 | 说明 |
|------|------|
| `ro` | 只读权限,可以浏览目录、查看文件内容、下载文件 |
| `rw` | 读写权限,可以浏览、上传、下载、删除文件 |

#### 4.4.3 用户结构

```c
typedef struct user {
    char username[32];              // 用户名
    char password_md5[32];          // 密码(MD5)
    permission_level_t perm;         // 权限等级
    struct user* next;              // 链表节点
} user_t;
```

#### 4.4.4 认证流程

```
1. 服务器启动 → 读取server.conf → 解析用户列表
2. 客户端连接 → 发送登录请求(username, password)
3. 服务器查找用户 → 验证密码 → 返回权限结果
4. 客户端根据权限使用相应功能
```

- 防止路径穿越攻击(检测`../`)

### 4.5 文件传输模块

#### 4.5.1 分块传输设计

大文件采用分块传输,避免占用过多内存:

```c
#define TRANSFER_BLOCK_SIZE  8192    // 每块8KB
#define MAX_CONCURRENT_TRANSFERS 3   // 最大并发传输数

// 传输任务结构
typedef struct transfer_task {
    uint32_t task_id;               // 传输任务ID
    char local_path[256];           // 本地文件路径
    char remote_path[256];           // 远程文件路径
    uint32_t file_size;             // 文件总大小
    uint32_t transferred;           // 已传输字节数
    uint8_t direction;              // 传输方向(上传/下载)
    uint8_t status;                 // 状态(等待/传输中/完成/失败)
    time_t start_time;              // 开始时间
    pthread_mutex_t lock;           // 任务锁
    struct transfer_task* next;     // 链表节点
} transfer_task_t;
```

#### 4.5.2 传输控制命令

| 命令 | 值 | 方向 | 说明 |
|------|------|------|------|
| CMD_FILE_UPLOAD_START | 0x0209 | C→S | 开始上传 |
| CMD_FILE_UPLOAD_DATA | 0x020A | C→S | 上传数据块 |
| CMD_FILE_UPLOAD_END | 0x020B | C→S | 上传完成 |
| CMD_FILE_DOWNLOAD_START | 0x020C | C→S | 开始下载 |
| CMD_FILE_DOWNLOAD_DATA | 0x020D | S→C | 下载数据块 |
| CMD_FILE_DOWNLOAD_END | 0x020E | S→C | 下载完成 |

#### 4.5.3 分块传输流程

```
上传流程:
客户端                        服务器
  │                             │
  │─── CMD_FILE_UPLOAD_START ──►│ 发送文件名、大小
  │◄─── ACK (task_id) ─────────│ 分配任务ID
  │                             │
  │─── CMD_FILE_UPLOAD_DATA ──►│ 发送第1块(8KB)
  │◄─── ACK ───────────────────│
  │─── CMD_FILE_UPLOAD_DATA ──►│ 发送第2块(8KB)
  │◄─── ACK ───────────────────│
  │         ...                 │
  │─── CMD_FILE_UPLOAD_END ────►│ 发送完成信号
  │◄─── ACK (status) ──────────│ 返回结果

下载流程类似,方向相反
```

#### 4.5.4 传输状态管理

```c
// 传输状态
typedef enum {
    TRANSFER_IDLE = 0,             // 空闲
    TRANSFER_WAITING = 1,           // 等待中
    TRANSFER_RUNNING = 2,           // 传输中
    TRANSFER_PAUSED = 3,            // 暂停
    TRANSFER_COMPLETED = 4,         // 完成
    TRANSFER_FAILED = 5             // 失败
} transfer_status_t;

// 传输进度计算
float get_transfer_progress(transfer_task_t* task) {
    return (float)task->transferred / task->file_size * 100;
}

// 传输速度计算
uint32_t get_transfer_speed(transfer_task_t* task) {
    time_t elapsed = time(NULL) - task->start_time;
    if (elapsed > 0) {
        return task->transferred / elapsed;
    }
    return 0;
}
```

#### 4.7.2 进程信息字段说明

| 字段 | 来源 | 说明 |
|------|------|------|
| PID | /proc/[pid]/stat | 进程ID |
| Name | /proc/[pid]/comm | 进程名称 |

| Memory | /proc/[pid]/status | 物理内存大小 |
| State | /proc/[pid]/stat | 进程状态 |
| Time | /proc/[pid]/stat | 累计CPU时间 |

#### 4.7.3 进程状态说明

| 状态 | 说明 |
|------|------|
| R | Running - 运行中 |
| S | Sleeping - 睡眠中 |
| D | Disk Sleep - 不可中断的睡眠 |
| Z | Zombie - 僵死 |
| T | Stopped - 已停止 |

#### 5.6.3 监控控制逻辑

**控制按钮说明**:
- **开始监控**: 启动定时器,同时刷新系统信息和进程列表
- **停止监控**: 停止定时器
- **刷新**: 立即刷新当前Tab的数据(系统信息或进程列表)
- **刷新间隔**: 设置定时刷新的间隔时间(1-60秒)

**监控行为**:
1. 点击"开始监控"后,两个Tab会同步刷新
2. 定时器触发时,同时更新系统信息和进程列表
3. 切换Tab不影响监控状态
4. 在任意Tab点击"刷新"只刷新当前Tab的数据
